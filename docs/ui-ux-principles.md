# Loopers Meetup UI/UX 원칙

이 문서는 화면 수정 시 반복해서 적용할 기준이다. 기능이 많아져도 사용자는 운영 구조를 이해할 필요가 없고, 화면은 지금 해야 할 일을 바로 보여줘야 한다.

## 1. 사용자 언어를 쓴다

- 내부 용어를 노출하지 않는다. 예: `운영 단위` 대신 `기수`.
- 권한 이름은 구분해서 쓴다. `전체관리자`는 기수 관리, `관리자`는 기수별 관리.
- 버튼은 섹션 맥락이 명확하면 짧게 쓴다. 예: `팀 추가` 대신 `추가`, `보고 수정하기` 대신 `수정`.
- 상태는 행동과 섞지 않는다. 예: `제출` 배지와 `수정` 버튼을 붙여 복잡하게 만들지 않는다.

## 2. 흐름이 끊기지 않게 한다

- 편집/상세 화면에는 목록으로 돌아가는 명확한 경로를 둔다.
- 목록에서는 항목 자체를 눌러 상세로 들어가고, 수정/삭제는 상세 안에 둔다.
- 저장 후에는 목록으로 돌아가거나 토스트로 결과를 알려준다.
- 로그아웃과 인증 만료는 항상 같은 첫 화면으로 돌아간다.

## 3. 화면 역할을 나눈다

- 첫 화면은 기수 선택만 담당한다.
- 입장 코드는 별도 입장 화면에서 입력한다.
- 전체관리자는 기수 생성/편집/코드 관리만 담당한다.
- 기수별 관리자/엔젤 기능은 전체관리자 권한에 의존하지 않는다.

## 4. 카드 남발을 피한다

- 반복 목록, 모달, 주요 편집 단위에만 카드를 쓴다.
- 하나의 리포트 내용은 여러 카드로 쪼개지 않고 문서처럼 보여준다.
- 항목 구분은 카드보다 타이포그래피, 여백, 구분선을 우선한다.
- 카드 안에 또 카드가 들어가는 구조를 피한다.

## 5. 정보는 중복하지 않는다

- 같은 의미의 숫자나 배지를 여러 개 보여주지 않는다.
- `제출 팀`, `미제출`, `보고`가 같은 상태를 말하면 `제출 현황` 하나로 줄인다.
- 멤버 수처럼 현재 판단에 필요 없는 정보는 숨긴다.
- 안내 문구는 사용자가 판단할 수 없는 예외 상황에서만 쓴다.

## 6. 토스트를 사용한다

- 생성/수정/삭제 완료는 화면 하단 중앙 토스트로 알린다.
- 완료 토스트는 대상 이름을 반복하지 않고 행동 결과만 짧게 쓴다. 예: `저장 완료`, `변경 완료`, `삭제 완료`, `취소`
- 완료 상태를 목록 상단 뱃지로 계속 남기지 않는다.
- 사용자가 다시 확인해야 하는 오류만 화면 안에 남긴다.

## 7. 폼은 라벨-본문 정렬을 맞춘다

- 입력 항목은 라벨과 입력 영역의 기준선을 맞춘다.
- 필수/선택 설정이 없다면 `필수` 배지를 보여주지 않는다.
- 템플릿 이름은 본문에 섞지 않고 배지로 분리한다.
- 긴 입력 폼은 카드 여러 개보다 한 덩어리의 행 구조로 구성한다.

## 8. 주간보고의 핵심만 남긴다

- 주간보고의 핵심은 주차 생성, 엔젤 작성, 제출 현황 확인이다.
- 팀과 엔젤 매핑은 멤버 관리가 기준이다.
- 댓글은 주간보고 범위에서 제외한다.
- 슬랙 공유 문구는 별도 기능으로 다루며 기본 화면 복잡도를 높이지 않는다.
